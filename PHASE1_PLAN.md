# AI 跑团助手 — Phase 1 实施计划

> 目标：用现有设备跑通"语音→识别→AI→显示"闭环，零硬件成本。

---

## 一、最低可行架构

```
[平板浏览器] ←→ WebSocket ←→ [PC Python Server]
                                  ├─ FunASR + CAM++ (语音转文字+声纹)
                                  ├─ DeepSeek API (AI DM 引擎)
                                  └─ 游戏状态管理 (JSON)
```

平板只做两件事：**收音**和**显示**。所有计算在PC。

---

## 二、分步实施

### Step 1：跑团 Agent（1-2天）
**只做文字，不碰语音。**

- 用 DeepSeek API + System Prompt 搭建 DM 角色
- System Prompt 包含：世界观设定、规则约束、叙事风格、NPC行为准则
- 写一个命令行交互脚本：用户打字 → DeepSeek 回话
- 验证点：AI 能否记住上下文、遵守规则、推动剧情而非被动应答
- 持续调 prompt，直到"感觉像个真正的 DM"

### Step 2：语音识别链路（2-3天）
**不连 AI，只验证语音。**

- 部署 FunASR（实时流式识别）+ CAM++（声纹辨认）
- 写一个测试脚本：录一段音频 → 输出"谁说了什么"
- 先单人测试，再双人测试
- 验证点：识别准确率、声纹区分能力、延迟是否可接受
- 注意：先用电脑麦克风测，不着急接平板

### Step 3：平板 Web 客户端（1-2天）
**不连 AI，只做录和显。**

- 一个极简 HTML 页面：录音按钮 + 对话气泡列表 + 游戏状态面板
- 平板浏览器打开 PC 的内网地址即可
- Web Audio API 收音 → WebSocket 发送 → 接收结果显示
- 验证点：平板能否稳定收音、延迟是否可接受、UI 是否直观

### Step 4：全链路整合（1-2天）
**三步连起来。**

- 平板录音 → PC 识别+声纹 → DeepSeek 生成回应 → 平板展示
- 加游戏状态面板：当前场景、角色状态、剧情进度
- 验证点：完整一次跑团对话闭环

---

## 三、技术选型细节

| 模块 | 方案 | 理由 |
|------|------|------|
| 语音识别 | FunASR paraformer-large | 中文最优，支持流式 |
| 声纹辨认 | CAM++ | 轻量，EER 0.89% |
| LLM | DeepSeek API | 百万上下文，成本低 |
| 后端 | Python FastAPI + WebSocket | 实时双向通信 |
| 前端 | 单页 HTML + Web Audio API | 零安装，平板浏览器即用 |
| 游戏状态 | JSON 文件 | 简单，后续换 SQLite |

---

## 四、不做什么（Phase 1）

- ❌ 不做硬件（ESP32/树莓派）
- ❌ 不做声纹数据库（Milvus），先内存比对
- ❌ 不做存档/读档
- ❌ 不做 TTS 语音播报
- ❌ 不做多设备同步
- ❌ 不做 UI 美化（功能优先）

---

## 五、验收标准

一条完整链路跑通：两个人在电脑前说话 → 平板显示谁说了什么 → DeepSeek 以 DM 身份回应 → 游戏状态面板更新。延迟可接受（<3秒总延迟），识别准确率可接受（>85%）。
